# ADR-009 — Record Types for Plugin Results (resolves C9)

- **Status:** Accepted (baseline for v1)
- **Date:** 2026-06-25
- **Related:** ADR-002 (C8 offset model), spec `04-type-system.md`, `udp_dns.nfdl`

## Контекст

При переписывании `udp_dns.nfdl` выявлен разрыв (C9): контекстно-зависимый
плагин (DNS pointer decompression) должен одновременно:
1. вернуть **декодированное значение** (строку имени), и
2. сообщить, **сколько байт он логически потребил** в локальном потоке
   (длина on-wire представления, переменная из-за компрессии).

Но по семантике §3.6 чистый `invoke` **не двигает локальный offset** (плагин
делает произвольный random-access по root-буферу, а не последовательное чтение).
Если бы `invoke` возвращал только `str`, у автора DSL не было бы способа
продвинуть offset на корректное число байт.

То же относится к будущим плагинам: `scan_crlf` (позиция + длина строки),
HPACK-декодер (значение + потреблённые байты), любой TLV-парсер в плагине.

## Решение

Расширить типовую систему **record-типами для результатов `invoke`** и разрешить
`.`-доступ к их полям (грамматика Postfix уже это допускает — `dec.wire_len`).

### Форма

```
let dec   = invoke("dns_decompress", __root_buffer, __root_offset);
//  dec : record{ name: str, wire_len: u16 }     (тип из манифеста плагина)
qname:    bytes[dec.wire_len];   // потребление выражено явно и видимо в DAG
```

### Свойства record-типа v1 (намеренно ограничен)

1. **Только как результат `invoke`.** Пользователь не объявляет record-типы в
   DSL напрямую (нет `struct`-синтаксиса в v1). Record — закрытая форма,
   порождаемая исключительно сигнатурой плагина из манифеста.
2. **Плоский, именованные поля.** `record{ f1: τ1, ..., fn: τn }`, где `τi` —
   скаляр, `bool`, `str`, `opaque` или `bytes` (без вложенных record в v1).
3. **Иммутабельный, по значению.** Привязывается только через `let`; поля
   читаются через `.`; запись невозможна.
4. **Типизирован статически.** Манифест плагина объявляет точный record-тип
   результата; `dec.wire_len` резолвится и типизируется на этапе typecheck.
   Доступ к несуществующему полю → `TypeError::NoSuchField`.
5. **Не slice-владелец.** Поле `str`/`bytes` в record следует обычным правилам
   владения: `str` — owned (плагин аллоцировал), `bytes`-поле record в v1
   **не поддерживается как zero-copy view** (избегаем висячих ссылок на
   plugin-временные буферы); если плагину нужно вернуть срез — он возвращает
   `wire_len`, а срез берётся из исходного буфера через `bytes[len]`.

### Манифест: объявление record-результата

```
plugin "dns_decompress" {
    purity = pure_stateless;
    args   = (opaque /*root_buffer*/, u64 /*root_offset*/);
    ret    = record{ name: str, wire_len: u16 };
}
```

Компилятор сверяет arity/типы вызова и типизирует `.`-доступы по `ret`.

## Последствия

- `04-type-system.md` включает `record{...}` в грамматику типов (только
  invoke-результат) и правило типизации Postfix `.field`.
- `udp_dns.nfdl` уже использует эту форму — становится валидным.
- Verifier: `dec.wire_len` участвует в field-DAG как обычная зависимость
  (`bytes[dec.wire_len]` зависит от `dec`, который зависит от `__root_offset`).
- Bounds: `wire_len` имеет тип `u16` → интервал `[0, 65535]`; `bytes[dec.wire_len]`
  получает runtime BOUNDS_CHECK против `__rem` (C5), т.к. точное значение
  известно только в рантайме.

## Отклонённые альтернативы

- **`invoke` двигает offset на скрытую величину** — нарушает «invoke pure, не
  трогает локальный offset», делает потребление невидимым в DAG, ломает
  воспроизводимость анализа границ. Отвергнуто.
- **Множественные возвращаемые значения через tuple-деструктуризацию**
  (`let (name, len) = invoke(...)`) — добавляет tuple-паттерны в грамматику без
  выигрыша над именованным record; имена полей самодокументируют. Отложено.
- **Полноценные user-defined struct-типы** — расширяет язык за пределы нужд v1
  (сообщения уже играют роль агрегатов). Кандидат на v2.
